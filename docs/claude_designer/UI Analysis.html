Summary
TL;DR

Strong vocabulary, four dialects

The cockpit has a confident aesthetic point of view — brutalist Quake-console, flag-inspired Pantheon/pTn palette, ticks-and-LEDs vibe. On the Review Console surface it delivers. The problem is that four separate design systems are running in the same application, and the other surfaces (Editor, Cinema, prototypes) don't speak the same language.

Nothing here is broken; the work-to-unify payoff is large and the changes are mostly mechanical: consolidate tokens, pick one type pairing per density tier, and remove the --orange → --gold shim so overrides stop fighting each other.

System fragmentation
Critical

Four stylesheets define four overlapping but non-identical systems. They disagree on color values, type, border radius, and semantic meaning of the same variable names.

Three distinct "gold" values in circulation

#f5a623 (ember-orange, closer to amber), #f5c518 (Nauru stripe, truer yellow-gold), #fbbf24 (Tailwind amber-400). These are visually distinguishable side-by-side. A user moving from Review Console → Editor will perceive the accent color shifting — reads as a different product.

Critical

The --orange → --gold shim is a landmine

pantheon.css aliases --orange: var(--gold) so that any style.css rule consuming --orange picks up gold. It works until someone writes a new rule referencing --orange without knowing the shim exists — they'll get gold on Review Console but orange (or nothing) on any surface that doesn't load pantheon.css. The alias also prevents you from using true orange anywhere deliberately, because the token name lies about what it holds.

Rename the token (--accent) once, remove the compatibility shim, sweep the codebase. This is a 20-minute job that eliminates a whole class of future bugs.

locus: pantheon.css:75–77

High

editor.css ignores the Pantheon brand entirely

Body font is Inter; the design system tokens are named --bg-0…--bg-3, --accent, --accent-alt. Border-radius is 3px. None of this is wrong in isolation — but it shares no vocabulary with the cockpit. A product with a strong brutalist identity on one route and a generic dark Tailwind feel on another reads as two products glued together.

locus: editor.css:1–15, 19 (font), editor.css:41 (3px radius)

02
Information architecture
Navigation
High

No global navigation between the four surfaces

Review Console (index.html) has no link to Editor or Cinema. Editor has a single ← Cinema Suite back-link. Cinema has none. The user either memorizes URLs or clicks around. There is no breadcrumb indicating where in the pipeline they are.

Suggest: a persistent top-strip with Review · Cinema · Editor as segmented control, current part ID, and render status LED carried across surfaces. Reuse the existing .part-nav component vocabulary.

loci: index.html:37–41, editor.html:21, cinema.html:11–18

Medium

Three-panel grid is rigid at 38% / 1fr / 24%

style.css:195 hard-codes the panel split. Below 1100px the layout stacks vertically (pantheon.css:714) but between 1100–1400px the Flow panel gets squeezed and the Event panel still keeps 24%. Users on laptops get a cramped middle panel that's where the densest information lives.

Suggest: resizable dividers (drag handles), or widen flow to 1.4fr with min-width guards on the outer panels.

locus: style.css:195

Medium

The three panels lack a task narrative

Compare · Flow · Events are spatially peers but they are not conceptually peers — Compare is the primary decision surface, Flow is the artifact being decided on, Events is evidence. Giving all three equal panel-bar treatment flattens the hierarchy.

Suggest: keep Compare visually dominant (full-width or larger), and treat Flow + Events as supporting rails underneath. Or at minimum, visually differentiate the primary panel with a heavier bar, larger title, or accent color.

03
Density & type scale
Typography

The cockpit pushes density hard, which fits the genre. But several type sizes cross into "not comfortably readable" territory.

High

7px and 8px text is used for meaningful labels

Legend entries (.leg, .ab-tag), LED status labels, thumbnail seam labels, hotkey indicators — all 7–8px. At a typical 96dpi display that's roughly 5.25–6px rendered height after the letter-spacing eats into tracking. This is below WCAG comfortable-read thresholds and below the game-console-chrome convention it's emulating (which was 10–12px at arm's length on a 27" CRT).

Suggest: set a 10px floor for any text carrying semantic meaning. Keep 7–8px only for decorative tick labels where meaning is carried elsewhere.

loci: style.css:269 (.ab-tag), pantheon.css:205 (logo-sub 8px), pantheon.css:541 (ruler-label 7px)

Medium

Three display fonts fighting for attention

Bebas Neue (hero), Black Ops One (heads), IBM Plex Mono (body), plus Inter on Editor. Bebas + Black Ops are both condensed stencil-ish display faces — they don't harmonize, they compete. Black Ops One in particular has a pronounced military-stencil personality that reads louder than Bebas and drags the tone toward "military game HUD" rather than "competitive esports console."

Suggest: pick one display face (Bebas Neue is the cleaner pick for pTn's clan identity) + IBM Plex Mono for data. Drop Black Ops One. Drop Inter from Editor; use Plex Mono there too.

locus: pantheon.css:26, 85–86

Low

Letter-spacing is aggressive across the board

Values of .22em–.32em are common on caps titles. On a mono face this is fine for 4–6 character pills but gets visually loose on longer strings like "REVIEW CONSOLE v2". Consider stepping down to .14–.18em above 10 characters.

04
Color & semantics
Tokens
High

Gold is overloaded — it means too many things

In pantheon.css, gold is simultaneously: the brand accent, the primary action color, the T1 tier marker, the "drop" section marker, the OK LED, the scrub fill, and the seam marker. When the same color means primary, elite, climax, approved, and structural seam at once, none of those signals are legible as signals.

Suggest: keep gold as brand accent only. Move the semantic meanings to: LED OK → green; Tier → desaturated gold gradient across tiers; Drop section → gold (the one exception); Seam → white or blue. This is more work but it's the right work.

loci: pantheon.css:96, 98, 100, 102, 227, 235, 545

Medium

Red is the only error signal but the palette has no warning tier

--led-err #c41515 exists. --led-warn is mapped to #c99a10 (dim gold) — which is indistinguishable from regular gold in peripheral vision. For a pipeline tool where "render degraded but shipped" and "render failed" are different states, they need different colors.

Suggest: introduce a true amber (#e68a00 or similar) between gold and red, distinct from both.

locus: pantheon.css:113

Medium

T3 tier is "near-invisible" on purpose — that's a bug, not a feature

--t3: #0d1228 sits ~2% luminance off the surface background. Clips in tier 3 effectively disappear on the flow timeline. If T3 isn't worth showing, don't show it; if it is, give it enough contrast to read. "Near-invisible" is a design decision that hides information.

locus: pantheon.css:104

05
Interaction gaps
UX
High

"DROP JSON" and "LOAD JSON" buttons are workflow smells

The comment in index.html:7–14 says JSON auto-loads when served from repo root, otherwise the user must manually drag-drop. Manual JSON loading is not a feature in a shipping tool — it's a dev-mode fallback. Surfacing it in the header chrome normalizes a broken state.

Suggest: detect the serve path at startup. If auto-load works, hide the buttons entirely. If it doesn't, show a single banner strip ("Dev mode — drop JSON to proceed") instead of putting two file-picker buttons in the persistent header.

locus: index.html:42–46

High

A/B compare has no keyboard scrubbing

The scrubber is mouse-only (div with click handlers, not an <input type=range>). Power users reviewing dozens of renders will want J/K/L, arrow keys, frame-by-frame stepping, and a mark-in/mark-out pair. None of that is present in the markup.

Suggest: replace .scrub-track with a styled native range input, bind arrow keys and J/K/L globally when Compare panel is focused. Add a hotkey legend — the prototype panels already have a .hotkey-legend component you can reuse.

loci: index.html:96–102, pattern already exists at pantheon.css:588

Medium

No empty states for the Flow or Events panels

Compare has a dashed drop-zone when empty. Flow and Events have nothing — the panel bar renders and below it is void. First-time users seeing a blank middle panel won't know whether the app is loading, broken, or waiting for input.

Suggest: reuse the drop-zone pattern. For Flow: "no flow plan loaded · drop flow_plan.json". For Events: "no event data · drop event_diversity.json". Include the small-format JSON-schema hint underneath.

Medium

Status LED is the only feedback channel for long operations

A 6px colored dot with a text label is fine for idle/ok/busy states. It is not fine for communicating "rendering chunk 7 of 42, ETA 4:12" which is exactly what the Editor and Cinema surfaces need during a ship.

Suggest: when status is busy with known progress, expand the LED region into a horizontal progress bar inline in the header. Keep the LED for discrete states only.

locus: style.css:175–181

Low

Hover-brightness is the universal hover treatment — it doesn't always work

.clip-block:hover { filter: brightness(1.4) } on already-bright gold pushes into near-white territory; on near-invisible T3 it does nothing perceptible. Hover needs to be a different channel from luminance in a palette this contrast-heavy.

Suggest: hover → add a 1px gold outline + preserve fill. Brightness filter for press states only.

locus: style.css:451

06
Accessibility
WCAG
High

Scanlines and low-contrast dim text fail WCAG AA in several spots

--text-mute #3a4478 on --bg #040610 is ~2.1:1 contrast — below AA for any text. The 3px scanline overlay at 7% opacity further lowers effective contrast across the whole surface. For a long-session tool this compounds into eye strain.

Suggest: two-pass fix. First, raise --text-mute to at least #5a6890 (~3.8:1) so it clears AA for large text. Second, gate the scanline overlay behind a "CRT mode" toggle in a settings panel rather than making it always-on.

loci: pantheon.css:69, pantheon.css:135–144

Medium

Custom buttons lack focus styles

.part-btn, .hdr-btn, .pnl-btn define hover but no :focus-visible. Keyboard users lose all indication of where focus is. In a tool that's ostensibly keyboard-friendly (it advertises hotkeys), this is a real gap.

Suggest: add a global :focus-visible outline (gold 1px dashed + 2px offset) and remove any outline:none declarations.

Low

Icon-only buttons have no accessible name

⊞ FLOW JSON pairs the glyph with a word so it's fine. But ▶ PLAY, ⇌ SYNC, − ZOOM / + ZOOM rely on glyph + short label — screen readers will read "dash zoom, plus zoom" which is borderline. Not a blocker, easy fix with aria-label.

07
Brand cohesion
Identity
Medium

The Pantheon/pTn identity is asserted in CSS comments but not in UI

The 20-line ASCII flag in pantheon.css is a lovely artifact. But the UI itself shows ⊕ PANTHEON // REVIEW CONSOLE v2 — no star, no flag stripe, no clan mark. The brand story exists in the comments, not on screen.

Suggest: consider a 1px gold horizontal stripe somewhere persistent (below header, above status bar) as a literal flag reference. Or a small 12-pt star glyph replacing the ⊕ cross. Small, quiet, gives the identity a visual home.

Low

"v2" in the logo-sub is a trap

Versioning in the persistent logo area means every major redesign needs the logo updated, or "v2" becomes a lie. Move the version indicator to the status area or a tooltip on the logo.

locus: index.html:33
